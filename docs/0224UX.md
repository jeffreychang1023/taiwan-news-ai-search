## Issue #1：「條列出12月10大新聞」

**回報日期**：2026/1/6
**功能**：新聞搜尋
**狀態**：已修復 (2026-02-25, branch: worktree-ux-fixes)

### 問題描述

- **A**：系統沒有列出 12 月十大新聞，而是搜到一篇「Yahoo 公布 2025 十大台灣新聞事件」的報導
- **B**：出現重複文章（同標題不同來源：經濟日報 vs 聯合新聞）

### 檢查結果（二次驗證）

- 問題 A 依舊：系統回傳「整年度十大新聞」文章，非 12 月十大
- 問題 B 依舊：相同標題文章重複出現（不同來源）

### 根因分析

#### 問題 A：搜尋意圖與呈現指令未分離

1. **QueryRewrite（已啟用）功能不足**：目前 prompt 只做「語意擴展」（抽象概念→具體新聞詞彙），沒有「拆分搜尋意圖 vs 呈現指令」的能力
2. **「條列出12月10大新聞」被整串當搜尋 query**：BM25 關鍵字匹配直接命中標題含「十大新聞」的文章
3. **上游 NLWeb 的 DetectQueryType 被註解掉**：`baseHandler.py:374-377`，原作者（R.V.Guha）在 `19baadf`（2025-09-02）大型重構時移除，commit message 僅 "more fixes"。Fork 後未重新啟用
4. **不重新啟用 DetectQueryType 的原因**：query 類型無限多（搜尋型、聚合型、比較型、追蹤型...），分類容易誤判，誤判會讓結果更差

#### 問題 B：重複文章 — 非 Bug

- MMR 基於 embedding cosine similarity 做多樣性，不是標題去重
- 同事件不同來源（經濟日報 vs 聯合新聞）是合理的多來源覆蓋
- 使用者可能正好想看不同來源的觀點差異
- 想看更多不同新聞的使用者，往下滾即可

**結論：問題 B 不需要修復。**

### 修復方向

**在 QueryRewrite prompt 中加入「呈現指令分離」規則**，不新增分類系統：

1. **修改 `config/prompts.xml` 的 QueryRewrite prompt**：
   - 新增規則：若 query 包含呈現指令（條列、比較、整理、歸納、排名...），從搜尋 query 中移除，只保留搜尋意圖
   - 例：`「條列出12月10大新聞」` → `rewritten_query: "2025年12月 重要新聞事件"`
   - 回傳結構新增 `display_instruction` 欄位保留原始指令

2. **讓 SummarizeResults 使用原始 query（含呈現指令）**：
   - Summarize prompt 看到「條列出十大」就會自然地從結果中提煉 Top 10 列表
   - 需確認前端在 `list` mode 時也能觸發 summarize（或自動切換 mode）

**風險評估**：低。最差情況是 rewrite 沒改好，搜尋結果與現在相同，不會更差。

## Issue #2：「台灣在12月中，政府有對治安政策有新的論述或是規劃嗎」

**回報日期**：2025/1/8
**功能**：新聞搜尋
**狀態**：已修復 (2026-02-25, branch: worktree-ux-fixes)

### 問題描述（首次測試）

- 模型給出立委兩岸關係、財劃法相關新聞，未找出張文隨機殺人事件後的治安規劃/應對措施

### 檢查結果（二次驗證，CEO 2026-02-24 重新測試）

測試 3 次（原始 query + 兩個變體），analytics DB 驗證：
- retrieved docs 全是政治/國防新聞（兩岸、財劃法、川普國安戰略、移工政策）
- **零治安相關文章進入候選集**
- LLM ranking 最高分僅 60 分（跨境鎮壓通報機制），多數 10-30 分
- 原始問題 B（搜尋失敗）未再重現，確認為暫時性 infra 問題

### 根因分析

#### 主因：Hybrid Search 的 Domain Filter 硬性過濾（嚴重 Bug）

**位置**：`retrieval_providers/qdrant.py:884-1137`

Query 「政府有對治安政策...」中的「**政府**」觸發了 domain indicator filter：

```python
domain_indicators = ['零售', '金融', '製造', '醫療', '教育', '政府', ...]
```

觸發後的行為：
1. `query_domains = ['政府']` — 偵測到「政府」domain
2. 每篇文章檢查標題是否含「政府」→ `has_critical_keyword`
3. 標題沒有「政府」的文章歸為 `off_topic_results`
4. **Line 1135-1137：只保留標題含「政府」的文章，其他全部丟棄**

結果：所有「治安」「犯罪」「刑案」相關文章（標題沒有「政府」）被濾掉，只留下標題含「政府」的政治/國防新聞。

#### 設計問題：Domain Filter 來自 Shopify 產品搜尋，不適用新聞搜尋

- `domain_entities` 裡全是零售公司（walmart、momo、7-eleven）
- 產品搜尋場景中「零售」filter 有意義（排除金融科技文章）
- 新聞搜尋場景中「政府」不代表「只看標題含政府的文章」

#### 語料補充

Qdrant 直接搜尋確認：「治安」= 0 篇、「犯罪」= 2 篇、「張文」= 1 篇。語料也有缺口，但即使有相關文章，domain filter 也會把它們濾掉。

### 修復方向

**將 domain filter 從「硬性過濾」改為「加分機制」**：

- domain indicator 匹配的文章 → 額外加分（boost）
- domain entity 匹配的文章（如 momo、walmart）→ 額外加分
- **不匹配的文章 → 保留，正常參與排序**
- BM25 + vector search 已經能處理相關性排序，domain boost 只是錦上添花

**具體修改**：`qdrant.py:1121-1137`，移除 `on_topic_results`/`off_topic_results` 分流邏輯，改為在 `final_score` 上加一個 `domain_boost` 係數。

**風險評估**：低。移除硬性過濾只會增加候選結果多樣性，不會讓好結果變差。

### 附加問題

- [ ] `num_results_retrieved` 在 queries 表中全部為 0，但 `retrieved_documents` 有 9-10 筆 — 計數 bug 待查
- [ ] MMR score 全部為 0.0，ranking_position 全部為 0 — 記錄 bug 待查

## Issue #3：「台灣在12月中，政府有對治安政策有新的論述或是規劃嗎」（深度研究版）

**回報日期**：2025/1/8
**功能**：深度研究
**狀態**：已修復 (2026-02-25, branch: worktree-ux-fixes)

### 問題描述（首次測試）

- **A-1**：深度研究報告中完全沒有治安相關內容
- **A-2**：報告中出現 `private://` 開頭的連結，使用者無法點擊
- **B**：模型將「12月」解讀為 2026 年 12 月（未來），而非 2025 年 12 月

### 檢查結果（二次驗證）

- A-1：同 Issue #2 根因（Domain Filter 硬性過濾），Analyst 收到的資料就已經沒有治安相關文章
- A-2：已修復（private:// 連結問題）
- B：年份誤判問題依舊

### 根因分析

#### 問題 A-1：同 Issue #2

Analyst 收到的 `formatted_context` 來自 Retrieval 階段，而 Domain Filter Bug 已在 Retrieval 階段將治安文章濾掉。Analyst 巧婦難為無米之炊。

**修復**：同 Issue #2（Domain Filter 改為 boost）。

#### 問題 A-2：已修復

不再追蹤。

#### 問題 B：年份誤判 — 完整追蹤

**調查路徑（三階段追蹤）**：

##### 1. TimeRangeExtractor（pre-retrieval）

`core/query_analysis/time_range_extractor.py`

- **Stage 1 (Regex)**：「12月中」不匹配 — `yyyy_mm` pattern 需要四位年份前綴（`\d{4}年\d{1,2}月`）
- **Stage 2 (LLM)**：有注入 `current_date`（line 328），LLM 收到 `Current date: 2026-02-24`。理論上能正確解析，但需 confidence >= 0.7 才採用（line 176）
- **Stage 3 (Keyword fallback)**：「12月中」不含 FALLBACK_KEYWORDS（最新、最近...）→ `is_temporal = False`
- **風險**：如果 Stage 2 LLM 失敗或 confidence 不足 → `is_temporal = False` → **不會加任何時間 filter**

##### 2. Retrieval 階段 — 時間 filter 有接上

`baseHandler.py:452-489`：

```python
temporal_range = getattr(self, 'temporal_range', None)
if temporal_range and temporal_range.get('is_temporal'):
    search_filters.append({"field": "datePublished", "operator": "gte", "value": start_date})
    search_filters.append({"field": "datePublished", "operator": "lte", "value": end_date})
```

Retriever **有使用** temporal_range 做 Qdrant date filter。但前提是 `is_temporal = True`。

##### 3. Analyst prompt（reasoning 階段）

`reasoning/prompts/analyst.py:300-474`：
- `_build_base_research_prompt()` **沒有** `current_date`
- 唯一有 `current_date` 的地方：`_build_mandatory_precheck()`（line 236），僅在 `enable_gap_enrichment AND enable_web_search` 都啟用時觸發

##### 4. Analytics 記錄問題

- `has_temporal_indicator` 在 290 筆查詢中**全部為 0** — 但這是 **logging bug**（`log_query_start()` 的 data dict 根本沒寫入此欄位），不代表功能失效
- `temporal_boost` 全部為 0.0 — 這是 Phase A placeholder，by design

**根因**：兩個階段都有潛在問題：

1. **TimeRangeExtractor**：「X月中」完全依賴 LLM Stage 2 解析。如果 LLM 失敗或 confidence < 0.7 → 不加時間 filter → Retrieval 返回任意時間文章
2. **Analyst prompt**：不知道今天日期 → 即使 temporal_context 正確傳入（`Time Range: 2025-12-X to 2025-12-Y`），Analyst 仍可能在報告正文中將「12月」寫成 2026 年

### 修復方向

**修復 1：Analyst prompt 注入 current_date**（優先）

```python
# reasoning/prompts/analyst.py 的 _build_base_research_prompt()
current_date = datetime.now().strftime("%Y-%m-%d")

# 加入 prompt：
f"""
**重要時間資訊**：
- 今天的日期：{current_date}
- 新聞只可能來自過去，不可能來自未來
- 若用戶提到月份但未指定年份，預設為最近的「過去」該月份
  （例：現在是 2026 年 2 月，「12月」= 2025 年 12 月，不是 2026 年 12 月）
"""
```

**修復 2：加強 Regex 覆蓋「X月中/X月初/X月底」**（次要）

在 `REGEX_PATTERNS` 新增 pattern，減少對 LLM Stage 2 的依賴：
```python
'month_period_zh': r'(\d{1,2})月[中初底末]'  # 匹配「12月中」「1月初」等
```

**修復 3：修復 logging bug**（附帶）

在 `log_query_start()` 或 `log_query_complete()` 中寫入 `has_temporal_indicator`。

**風險評估**：修復 1 極低（最差情況 Analyst 忽略，與現狀相同）。修復 2 低（regex 是確定性的，不會讓非時間查詢誤判）。

```
相關全域 state 變數：
```

#### 問題 D：摘要語言從中文變英文（語言一致性問題）

**調查結果**：語言設定散落各處，缺少統一 guardrail：

| 位置 | 設定 | 問題 |
|------|------|------|
| `writer_templates.yaml:182` | `default_language: zh-TW` | ✅ |
| `reasoning/prompts/*.py` | 整份 prompt 用繁中 | ✅ |
| `config/prompts.xml` | 「respond in same language as query」 | ⚠️ 依賴 query 語言，非強制中文 |
| `llm_providers/anthropic.py:119` | 英文 system prompt | ❌ 無語言指示 |

**根因**：沒有「必須用繁體中文」的硬性約束。`prompts.xml` 只說「用和 query 相同語言回應」，如果 query 被 rewrite 時混入英文，或 LLM 隨機傾向英文，就會切換語言。

**修復方向**：在所有 prompt entry point 加入明確語言約束：

1. `prompts.xml` 的 SummarizeResults / RankItem prompt — 加入 `「必須使用台灣用語的繁體中文回應」`
2. `llm_providers/anthropic.py` 的 system prompt — 加入中文輸出指示
3. Reasoning agent prompts — 已是中文但可加強硬性約束

**風險評估**：修復 C 中等（前端 state 管理需全面審視，改動範圍較大）。修復 D 低（加語言約束不影響功能）。

## Issue #6：「今日的股市趨勢」

**回報日期**：2026/1/10
**功能**：新聞搜尋
**狀態**：已修復 (2026-02-25, branch: worktree-ux-fixes)

### 問題描述（首次測試）

- 摘要是今日的，但搜尋結果是 12 月底的

### 檢查結果（二次驗證，2026-03-23）

- 回應顯示 12 月份的股市資訊（不只一天），僅顯示其中幾天
- 使用者問「今日」，模型應知道今天日期，若無符合新聞應明確告知

### 根因分析

#### Retrieval 行為 — 合理的 fallback

TimeRangeExtractor 的 regex 有 `today_zh: r'今天'`，「今日」可能不匹配（需確認），但即使正確偵測到「今天」並加上 date filter，Qdrant 裡沒有當天的新聞時，系統會 fallback 到無時間 filter 的搜尋，返回最接近且相關的結果（12 月底股市新聞）。

**這個 fallback 行為本身是合理的** — 不應因為今天沒新聞就直接放棄搜尋。

#### 摘要未反映實際時間差距 — 真正的問題

SummarizeResults prompt 會注入大量 12 月份的新聞作為 context，LLM 被這些內容 skew，直接基於 12 月資料生成摘要，沒有意識到（或沒有告知使用者）「這些不是今天的新聞」。

**根因**：SummarizeResults prompt 缺少以下資訊：
1. 使用者原始 query 中的時間意圖（「今日」）
2. 實際搜尋結果的時間範圍（12 月底）
3. 兩者之間的落差

LLM 只看到一堆 12 月新聞 + 「股市趨勢」query，自然就寫出 12 月的股市摘要。

### 修復方向

**在 SummarizeResults prompt 中注入時間落差提示**：

當 TimeRangeExtractor 偵測到時間意圖、但實際搜尋結果的日期與目標時間不符時，在 prompt 中加入：

```
使用者詢問的時間：今日（2026-01-10）
實際搜尋結果的時間範圍：2025-12-25 ~ 2025-12-31
注意：搜尋結果與使用者指定的時間不符。請在摘要中明確告知使用者，
目前資料庫中沒有符合指定時間的新聞，以下是最接近的相關結果。
```

這樣 LLM 就會生成「目前沒有今天的股市新聞，以下是最近一期（12月底）的相關報導」之類的摘要。

**附帶修復**：確認 `today_zh` regex 是否涵蓋「今日」（目前只有 `r'今天'`）。

**風險評估**：低。最差情況是 LLM 忽略時間落差提示（與現狀相同）。正常情況下會大幅改善使用者體驗。

## Issue #7：「2025/9以前的 iThome記者：郭又華 的文章」

**回報日期**：2026/1/12
**功能**：新聞搜尋
**狀態**：已修復 (2026-02-25, branch: worktree-ux-fixes)

### 問題描述

- 搜尋特定記者的文章，系統未正確識別記者名稱，無法以 author 欄位過濾

### 檢查結果（二次驗證）

- 系統未能正確提取記者名稱進行 author 過濾

### 根因分析

#### AuthorIntentDetector 已存在且已整合，但 regex 有 bug

**背景**：此問題在 `docs/archive/plans/phase5-investigation.md` Bug #21 中記錄過，當時被 Phase 3 Bug #18-20（通用 filter 架構）阻擋。Phase 3 已完成，AuthorIntentDetector 已整合至 `baseHandler.py:358`（pre-check）和 `baseHandler.py:491-497`（filter injection）。

**但 regex pattern 有嚴重 bug**：

`core/query_analysis/author_intent_detector.py:30-40`

| Pattern | Regex | 測試 query | 提取結果 | 正確？ |
|---------|-------|-----------|---------|--------|
| `author_articles_zh` | `(?:記者\|作者\|編輯)\s*([^\s的寫]{2,8})\s*(?:的文章\|...)` | `iThome記者：郭又華 的文章` | `：郭又華`（含冒號） | ❌ |
| `reporter_name_zh` | `(?:記者\|作者)\s*([^\s]{2,8})` | `王宏仁記者有什麼文章?` | `有什麼文章?` | ❌ |
| `reporter_name_zh` | 同上 | `王若樸記者在2025 9月之前` | `在2025` | ❌ |
| `by_author_zh` | `([^\s]{2,8})\s*(?:記者\|作者\|編輯)` | `王宏仁記者有什麼文章?` | `王宏仁`（正確） | ✅ |

**問題**：
1. `author_articles_zh`：`[^\s的寫]` 沒排除標點符號（冒號、：等），導致冒號被捕獲
2. `reporter_name_zh`：pattern 是 `記者\s*(.*)` 的變體，名字在「記者」**前面**時會捕獲記者後面的文字
3. `by_author_zh`：能正確捕獲「名字+記者」格式，但排在第三順位，被前面的 pattern 搶先匹配
4. **名字必須是純名字**：提取結果應為 `王宏仁` 而非 `王宏仁記者` 或 `王宏仁副總編輯`，因為 author 欄位中的名字不一定帶職稱

### 修復方向

**重寫 AUTHOR_PATTERNS 中文部分**：

核心原則：**提取純名字（2-4 個中文字），不含職稱、標點、後綴**

```python
AUTHOR_PATTERNS = {
    # 1. 「名字+記者/作者/編輯」 — 最常見的中文表述（優先）
    'name_before_title_zh': r'([\u4e00-\u9fff]{2,4})\s*(?:記者|作者|編輯|副總編輯|總編輯|主筆)',

    # 2. 「記者/作者：名字」或「記者/作者 名字」 — 帶分隔符
    'title_colon_name_zh': r'(?:記者|作者|編輯)[：:]\s*([\u4e00-\u9fff]{2,4})',

    # 3. 「名字 的文章/的報導」 — 所有格表述
    'name_possessive_zh': r'([\u4e00-\u9fff]{2,4})\s*(?:的文章|的報導|寫的|的新聞)',

    # English patterns (unchanged)
    'articles_by_en': r'...',
    'by_author_en': r'...',
    'reporter_en': r'...',
}
```

**關鍵改進**：
- 用 `[\u4e00-\u9fff]{2,4}` 限制中文字元 2-4 字（中文名字範圍）
- `name_before_title_zh` 排第一：「王宏仁記者」→ 提取 `王宏仁`
- `title_colon_name_zh` 處理冒號分隔：「記者：郭又華」→ 提取 `郭又華`
- `name_possessive_zh` 處理所有格：「郭又華的文章」→ 提取 `郭又華`

**風險評估**：低。regex 改進只影響 author 提取準確度，不影響其他搜尋路徑。改壞最差情況 = 現狀（提取失敗，fallback 到正常搜尋）。

### 架構決策：統一 QueryUnderstanding（Issue #7-9 討論後決定）

Issue #7-9 的 regex bug 討論後，發現更根本的架構問題：**query 分析邏輯散落三處，且純 regex 方案不可靠**。

#### 現況問題

```
query 分析散落地圖：
① core/query_analysis/          ← pre-retrieval（結構化）
   ├── TimeRangeExtractor        → regex + LLM（有兜底）
   └── AuthorIntentDetector      → regex only（壞掉中，唯一無 LLM 兜底的）
② retrieval_providers/qdrant.py  ← retrieval 內嵌（ad-hoc）
   ├── domain_indicators          → hardcoded string match（Issue #2 根因）
   ├── domain_entities            → hardcoded company list
   └── temporal_keywords          → hardcoded keyword list
③ reasoning/prompts/clarification.py ← Deep Research only
```

被註解掉的 5 個原作者分析器（DetectItemType, DetectQueryType 等）全部都是 LLM-based，AuthorIntentDetector 是整個系統中唯一純 regex 無 LLM 兜底的分析器。

#### 決策：合併為統一 QueryUnderstanding

**新增** `core/query_analysis/query_understanding.py`，合併以下功能為單一 LLM call：

| 原本 | 位置 | 合併後 |
|------|------|--------|
| QueryRewrite（語意擴展） | query_rewrite.py | → `rewritten_queries` |
| AuthorIntentDetector（記者偵測） | author_intent_detector.py | → `author` |
| TimeRangeExtractor Stage 2（LLM） | time_range_extractor.py | → `time_range` |
| Domain detection（hardcoded） | qdrant.py:884-898 | → `domain` |
| 呈現指令分離（Issue #1） | 不存在 | → `display_instruction` |

**Pre-retrieval 流程變化**：

```
現在（4-5 個 LLM call）：         改後（2 個 LLM call）：
① Decontextualize → LLM         ① Decontextualize → LLM（不變）
② QueryRewrite → LLM（等 ①）    ② QueryUnderstanding（等 ①）
③ TimeRange → regex → LLM          ├── Regex fast path（time + author）
④ Author → regex only（壞掉）      └── 單一 LLM call → 全部輸出
⑤ ToolSelector → LLM            ③ ToolSelector → LLM（不變）
⑥ Memory → DB                   ④ Memory → DB（不變）
```

**Regex 保留為 fast path**：明確的時間格式（`2025年12月`）regex 秒殺，不問 LLM。

**向後相容**：QueryUnderstanding 設定相同的 handler 屬性（`rewritten_queries`, `temporal_range`, `author_search`），下游程式碼不需要改。

**qdrant.py 改動**：移除 hardcoded domain_indicators/entities，改讀 `handler.domain_context` 做 boost（非硬過濾）。

**一次解決 Issue #1、#2、#3、#7-9**。

### 架構發現：偽 Hybrid Search

測試過程中發現系統的 BM25 是 **re-ranker 而非獨立 retriever**：

```
現狀（偽 hybrid）：
Query → Embedding → Qdrant vector search (top 500) → Python BM25 re-rank → output

正確架構（真 hybrid）：
Query ──┬── Dense (vector)  → N results ─┐
        └── Sparse (keyword) → N results ─┤→ Fusion → final ranking
```

當搜尋 keyword-heavy query（作者名、法案名等）時，如果目標文章不在 vector top-500 中，BM25 完全無用。此為作者搜尋失敗的根本原因。

**詳細分析**：`docs/plans/hybrid-search-architecture.md`
