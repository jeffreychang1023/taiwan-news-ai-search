## 會議 1：報告導讀（2026-02-13）

### CEO 修正：多租戶隔離模型

研究報告原本假設「每個租戶看到不同新聞」，CEO 澄清實際需求：

| 資源 | 共享 or 隔離 | 說明 |
|------|-------------|------|
| **新聞資料庫** | ✅ 共享 | 所有租戶搜同一份新聞，這是核心價值 |
| **對話紀錄** | 🔒 隔離 | 每家公司只看到自己的搜尋/對話歷史 |
| **搜尋歷史** | 🔒 隔離 | 同上 |
| **上傳資料** | 🔒 隔離 | 租戶自行上傳的文件/資料 |
| **連結的 DB** | 🔒 隔離 | 租戶自行連結的外部資料庫 |
| **LLM Model** | ✅ 共享 | 目前無 model selector，所有租戶用同一模型 |
| **用量配額** | 🔒 隔離 | 不同 tier 有不同用量上限 |

**影響**：這比原始研究假設的「資料隔離」簡單得多。Qdrant 新聞 collection 不需要 tenant_id filter，只有 conversation/history 等 user-generated 資料需要隔離。

### 成本總覽

| 規模 | 基礎設施/月 | LLM API/月 | 總計/月 |
|------|-----------|-----------|--------|
| 小（10 家客戶） | ~$100 | ~$630-900 | **~$800-1,000** |
| 中（50 家客戶） | ~$340-500 | ~$3,150-4,500 | **~$3,500-5,000** |
| 大（200 家客戶） | ~$680-1,200 | ~$12,600-18,000 | **~$13,000-19,000** |

> CEO 備註：基礎設施細部拆解待後續討論時詳細說明。如有更新需回頭修正此表。

### 關鍵洞察

- LLM API 佔 TCO 的 85-97%，基礎設施相對次要
- 成本控制重點在 LLM caching / model tiering，不在雲端選型
